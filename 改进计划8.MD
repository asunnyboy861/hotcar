# 改进计划8：数据流架构统一优化

## 1. 需求描述

### 1.1 背景
当前软件多个模块使用不同的数据流模式，导致：
- 部分页面数据更新不及时（如保养提醒添加后不立即刷新）
- 存在冗余的 Notification 通知机制，不可靠且难以维护
- 数据流不一致，增加开发和调试成本

### 1.2 目标
统一所有模块的数据流模式，实现：
- ✅ 数据变更实时响应，无延迟刷新
- ✅ 移除冗余的 Notification 机制
- ✅ Service 层 @Published 作为唯一数据源
- ✅ 与现有 Vehicle 模块数据流模式完全一致
- ✅ 平滑过渡，不影响现有功能

### 1.3 范围
- Vehicle Management 模块（已部分实现）
- Maintenance Reminders 模块（待完善）
- Settings 模块（可选）
- Statistics 模块（可选）

## 2. 生成规则

### 2.1 架构规范
所有模块必须遵循以下数据流模式：
```
┌─────────────────┐
│   Service Layer │
│  @Published     │  单例，唯一数据源
└─────────────────┘
          ↓ Combine 订阅
┌─────────────────┐
│  ViewModel Layer│
│  @Published     │  业务逻辑处理
└─────────────────┘
          ↓ @StateObject
┌─────────────────┐
│    View Layer   │
│  SwiftUI View   │  UI 渲染
└─────────────────┘
```

### 2.2 Service 层规范
```swift
@MainActor
final class XxxService: ObservableObject {
    static let shared = XxxService()
    
    // MARK: - 唯一数据源

        loadItemsSync()

    // MARK: - CRUD 方法

    func updateItem(_ item: XxxItem) async {

        saveItems()
    }

        // 保存到 UserDefaults/数据库

```

### 2.3 ViewModel 层规范
```swift
@MainActor
final class XxxListViewModel: ObservableObject {
    @Published var items: [XxxItem] = []
    
    private let service = XxxService.shared
    private var cancellables = Set<AnyCancellable>()
    private let itemId: String // 如 vehicleId 等过滤条件
    
    init(itemId: String) {
        self.itemId = itemId
        setupBindings()
        // 初始加载一次数据，确保首次显示正确
        loadInitialData()
    }
    
    // MARK: - Combine 绑定
    private func setupBindings() {
        service.$items
            .receive(on: RunLoop.main)
            .sink { [weak self] items in
                guard let self = self else { return }
                // 按业务需要过滤/排序，例如按 vehicleId 过滤
                self.items = items.filter { $0.relatedId == self.itemId }
                    .sorted { $0.date ?? .distantPast < $1.date ?? .distantPast }
            }
            .store(in: &cancellables)
    }
    
    // MARK: - 初始数据加载
    private func loadInitialData() {
        // 首次加载时同步一次数据，确保与服务层一致
        items = service.getItems(for: itemId)
    }
    
    // MARK: - 业务方法
    func addItem(_ item: XxxItem) async {
        await service.addItem(item)
    }
}
```

### 2.4 View 层规范
```swift
struct XxxListView: View {
    @StateObject private var viewModel = XxxListViewModel()
    
    var body: some View {
        List(viewModel.items) { item in
            XxxItemRow(item: item)
        }
        // ❌ 禁止使用 onReceive(Notification)
        // ❌ 禁止手动调用 loadItems() 刷新
    }
}
```

## 3. 检验标准

### 3.1 代码质量检查
| 检查项 | 标准 | 示例 |
|--------|------|------|
| Service 单例 | 必须使用 `static let shared` 单例模式 | ✅ `static let shared = MaintenanceService()` |
| @Published | 必须有且仅有一个 @Published 数据源 | ✅ `@Published var reminders: [MaintenanceReminder] = []` |
| ViewModel 初始化 | 必须接收过滤参数（如 vehicleId） | ✅ `init(vehicleId: String)` |
| Combine 绑定 | ViewModel 必须在 init 中 setupBindings() | ✅ 参考 VehicleListViewModel |
| 异步方法 | 所有 CRUD 方法必须是 async | ✅ `func addReminder(_ reminder: MaintenanceReminder) async` |
| 过滤逻辑 | 必须在 Combine sink 中完成过滤和排序 | ✅ `items.filter { $0.vehicleId == self.vehicleId }` |
| 兼容过渡 | Notification 标记为 deprecated 后再移除 | ❌ 禁止直接删除正在使用的 Notification |

## 5. 代码复用原则

### 5.1 复用现有模式
完全复用 Vehicle 模块已验证的 Combine 架构，不需要重新设计。
- ✅ 复用 VehicleListViewModel 的 Combine 绑定模式
- ✅ 复用 VehicleService 的 @Published + 单例模式
- ✅ 复用相同的异步方法签名规范

### 5.2 抽象原则
当前已有 2 个模块（Vehicle, Maintenance）使用该模式，未达到"三次法则"（需要3次及以上重复场景才抽象），暂不进行基类抽象。
当有 3 个及以上模块使用相同模式时，再考虑抽象 `BaseListViewModel` 通用基类。

### 5.3 与现有代码的兼容性
- Vehicle 模块已经完全符合该架构，无需修改
- Maintenance 模块只需添加 Combine 绑定，现有 API 保持不变
- 所有现有调用方式保持兼容，无需修改上层代码
